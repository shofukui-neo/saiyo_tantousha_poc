究極の営業リスト作成アプリ — システム概要 & データフロー

企業選定（発掘）→ 名寄せ → 公式URL → 電話 / 採用担当者 → メール → Tier → 担当者マスタ出力 を 1コマンドで実行する一気通貫パイプライン（Node.js / ローカル動作）
エントリ: node src/app.js 言語: Node.js (CommonJS) 依存: cheerio / playwright / googleapis / dotenv APIキー無しでも全工程が動作 キー投入で高精度経路に自動点火

本資料は poc/src/ 配下の実装実体（全モジュール）を走査して作成した、現状のシステム概要図とデータフローのまとめです。設計思想は「キー無しでも動き、キーがあれば点火する」— 各レイヤは利用可能な API キーに応じてローカル経路 / 高精度経路を自動で切り替えます。

2. 1 社の処理フロー（pipeline.processCompany() の内部）

候補 1 社が司令塔に入ってから「担当者マスタ 1 レコード」と「集計用 result」が出るまでの順序。

1
法人番号の名寄せ（任意）— nta.js。NTA_APP_ID があり、かつ法人番号が未設定なら商号で名寄せ。所在地も補完。
2
gBizINFO で属性補完（任意）— gbiz.js gbizGet。代表者名・公式HP・所在地（＝同名取り違え防止の addressHint）を取得。
3
公式URLの確定 — 入力/補完が無ければ search.js discoverUrl。検索→候補スコアリング→上位を実取得して社名＋所在地一致を検証。未発見なら NO_URL で終了。
4
robots 判定 — robots.js。トップが Disallow なら SKIP_ROBOTS で終了。
5
巡回ページの決定 — トップHTMLから discoverPages（ナビ）＋ sitemap（structured）＋ guessContactPaths ＋ LOCATE_PATHS を結合し重複除去、MAX_PAGES_PER_SITE（既定8）まで。
6
各ページをループ取得・抽出 — fetch.js（静的→Playwright）。各ページで ①JSON-LD telephone → ②正規表現で電話（phone.js、会社概要系ページは加点）、③採用担当者（recruiter.js）。担当者HIT かつ 電話確定で break。
7
メール推定 — email.js enrichEmail（MX篩い＋パターン、任意でHunter検証）。
8
finalize — score.js で Tier 判定・架電呼称生成・所要時間確定 → レコードを返す。
採用担当者の検証ゲート（validate.js）： 抽出結果は ①人名らしさ（会社名/部署/役割語でない）②採用/人事ロール語の近接 ③確度 ≥ CONFIDENCE_THRESHOLD（既定0.6）の全通過時のみ HIT。AI 経路・正規表現経路のどちらを使っても同じゲートを通します。

3. 発掘層の 2 経路（企業名そのものの収集）

gBizINFO 構造化発掘 GBIZ_TOKEN

discovery.js · discoverViaGbiz / gbiz.js
  • 都道府県 × 業種(business_item) でページネーション検索（株式会社/合同会社に限定）
  • 業種KW部分一致（事業概要＋営業品目）・設立年・従業員数レンジで選別
  • 補助金採択フラグ（source=4 突合）を任意付与
  • discoveryIcpScore（0–100）で降順ソート

Bing 発掘（API不要） 既定

discover.js · discoverCompanies / search.js
  • キーワード（業種・地域）でBing検索 → タイトル/スニペット＋上位ページ本文を抽出
  • または企業一覧ページURL（会員名簿等）の構造要素から抽出
  • 法人格（株式会社…）の前後関係＋助詞除去で社名コアを一意化（過剰一致抑制）
  • 社名コアで重複除去。確信が持てない断片は不採用（精度優先）

※ app.js --discover "<KW>" / --discover-url <URL> で明示クエリを渡した場合は、トークンの有無に関わらず Bing 経路が優先されます。

8. 設計原則（横断的に効いている方針）

① キー無しでも動く

全レイヤがローカル経路を持ち、*Available() 判定で自動分岐。コード変更なしでキー投入が効く。

② 礼儀正しいクロール

robots.js 遵守・POLITE_DELAY_MS・説明的UA・媒体内は1社ずつ。対象は公開ページのみ。

③ 精度優先（ゴミを流さない）

検証ゲート・社名/住所一致・確信できない断片の不採用で、後段に誤データを渡さない。

④ 構造化データ最優先

JSON-LD telephone > tel:リンク > 正規表現の順。sitemap でページ発見を補強。

⑤ 同名取り違え対策

gBiz/NTA の所在地を addressHint にして URL 検証で都道府県＋市区一致を要求。

⑥ 結果を失わない

CSV は常に出力。バッチは再開ジャーナル。Sheet 書き戻し失敗時も CSV は残る。
