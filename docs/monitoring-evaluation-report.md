# 新卒採用リアルタイム鮮度モニタリング — 実現可能性・効果測定・改善プロセス 報告書

作成: 2026-06-18 ／ 対象: `poc/src/monitor/`（設計は [monitoring-design.md](monitoring-design.md)）

---

## 0. エグゼクティブサマリ

**結論：実現可能。ただし2つの設計上の穴を塞ぐことが前提条件。**

| | 判定 | 根拠 |
|---|---|---|
| 技術的実現性 | ◯（条件付き） | 求人ボックスを静的取得で実証済（1サイクル24社/49件取得・所要約5秒）。他媒体はPlaywright経路で拡張可だがbot検知の壁あり |
| 鮮度の取得 | ◎（未活用の好材料あり） | 求人ボックスHTMLに「新着／本日／2日前／20日前」の**実掲載日マーカーが存在**。現状未使用＝最大の改善余地 |
| 法務・倫理 | ◯（要遵守設計） | robots遵守・レート制限は実装済。著作権法30条の4（情報解析）が根拠。媒体規約は個別精査必要 |
| 運用 | ◯ | 状態ディスク永続で中断再開可。cron/常駐どちらも対応 |
| コスト | ◎ | ローカル処理でAPI費ゼロ。Gemini任意 |

### 🚨 着手前に塞ぐべき2つの穴（Blocker）
1. **キャッシュTTL(7日) × サイクル間隔の相互作用**：`polite.js` の既定キャッシュは7日。同一クエリを60分毎に叩いても**同じキャッシュHTMLが返り差分が常に0化する**＝real-time監視が成立しない。
   実走の2サイクル目で「東京＝差分0、新規の大阪クエリ＝20社検知」となったのは、この副作用が原因（大阪は別URLでキャッシュ未ヒットだったため取れただけ）。
   → **対策**：監視経路は `SCRAPE_CACHE_TTL_MS` をサイクル間隔の半分以下に設定、または条件付きGET（ETag/Last-Modified）を `polite.js` に追加。後述4.1。
2. **擬似鮮度問題**：現状の `NEW` は「我々が初めて観測した」であり「実際に新しく掲載された」ではない。クエリを増やすほど“初観測”が量産され、鮮度ランキングが観測網の都合で歪む。
   → **対策**：カードの掲載日マーカー（◯日前）をパースして**真の掲載鮮度**を熱量に組み込む。後述4.2。

この2点を塞げば、本システムは「今アツい新卒採用企業」を実用精度で常時ランキングできる。

## 1. 実現可能性の精査

### 1.1 技術的実現性
| 観測ソース | 取得方式 | 状態 | リスク |
|---|---|---|---|
| 求人ボックス | 静的HTML（cheerio） | **実証済**・既定ON | セレクタ劣化（DOM変更） |
| リクナビ/マイナビ/ワンキャリア | Playwright（JS描画） | 掲載有無判定は実装済、観測器化は未 | bot検知・規約・低速 |
| Indeed | — | 403で無効化済 | 取得不可 |
| ハローワーク/Wantedly | 既存層あり（[[name-acquisition-layer]]） | 観測器化は未 | — |

- **セレクタ劣化**が最大の技術リスク。媒体はDOMを頻繁に変える。検知（全クエリ0件で `selector-stale`）は実装済、自己修復（Gemini再導出）はフックのみ。
- スループット：polite の直列＋4秒間隔がボトルネック。1クエリ1ページ約5秒。MAX_QUERIES=12で1サイクル約1分。これは礼儀（負荷抑制）のための意図的設計であり、監視網を広げるほどサイクル長が伸びるトレードオフ。

### 1.2 法務・倫理
- **適法性の根拠**：日本の著作権法30条の4（情報解析目的の複製は権利者の許諾不要）。robots.txt遵守・レート制限・キャッシュで重複取得抑制を実装済。
- **規約リスク**：各媒体の利用規約にスクレイピング禁止条項がある場合、債務不履行・不正競争防止法上の論点が生じうる。求人ボックスはrobotsで許可された範囲を取得。**リクナビ/マイナビ等の媒体直叩きを本番投入する前に各社規約の個別精査が必須**（法務確認をプロセスに組み込む／後述5.4）。
- 取得は公開情報・会社単位の事実（求人数・職種・掲載日）に限定し、個人情報（採用担当者名）の扱いは既存層の方針（[[recruiter-name-segment-finding]]）に従う。

### 1.3 運用・コスト
- 状態（スナップショット/熱量/クエリ）は `data/monitor/` に全永続。プロセスが落ちても次サイクルから継続。
- 常駐(`--watch`)よりcron/タスクスケジューラで`--once`定期起動が**耐障害性で推奨**（1サイクルが独立、メモリリーク・ハング耐性）。
- コスト：ローカル処理のみでAPI課金ゼロ。Gemini自己修復は任意点火。

## 2. 効果測定：テスト方法と指標（3階層）

価値検証は「①正しく動くか → ②鮮度を当てているか → ③売上に効くか」の3階層で分離して測る。下の層が崩れると上の層の数字が無意味になるため、順に固める。

### 階層① 機能テスト（決定論・無料・毎コミット）
**目的**：観測→差分→熱量→出力の各モジュールが仕様通り計算するか。
**方法**：合成スナップショット（固定JSON）を入力し、`diff`/`heat`/`rank` の出力をアサート（ネットワーク非依存のユニットテスト `test/monitor.test.js` を新設）。
| 指標 | 定義 | 合格基準 |
|---|---|---|
| 差分正当性 | 既知の前後スナップショットから期待イベント列が出るか | 100%一致 |
| 熱量再現性 | 同入力で熱量・順位が決定論的に再現 | 完全一致 |
| 減衰正当性 | 半減期72hで heat が `0.5^(h/72)` になるか | 誤差<1% |
| 暴走バックストップ | クエリ増殖が上限(64)で停止 | 超過しない |

### 階層② 鮮度精度（要・実掲載日。4.2実装が前提）
**目的**：ランキング上位が「本当に今アツい＝最近掲載/更新された」社か。
| 指標 | 定義 | 目標 |
|---|---|---|
| 検知リードタイム | 実掲載日 → 我々の検知時刻の差 | 中央値 < 24h |
| 鮮度的中率 | top-N社のうち実掲載が直近7日以内の割合 | > 70% |
| 偽差分率(false NEW/GONE) | 掲載内容不変なのに検索順位変動でNEW/GONE誤発火した割合 | < 10% |
| 観測網羅率 | （実在新卒求人数の推定 / 観測社数）。サンプル監査で推定 | トレンド改善 |

**バックテスト**：過去に大量採用を公表した既知企業数社について、公表日前後で熱量が先行スパイクしたかを検証（snapshots履歴を遡及再計算）。

### 階層③ 下流価値（最重要・要 outcomes.csv）
**目的**：熱量ランキングが実際の営業成果（接続→アポ→商談→受注）を予測するか。**既存 `source-kpi.js` の評価機構をそのまま流用**（[[multi-source-merge-kpi]]）。
| 指標 | 定義 | 判定 |
|---|---|---|
| 熱量帯別アポ率 | 熱量 高/中/低 帯ごとの アポ率・接続率 | 高帯 > 低帯 で単調なら有効 |
| リフト | 高熱量top-N架電のアポ率 ÷ 従来リスト(daihyou-1000等)のアポ率 | > 1.3 を目標 |
| 鮮度→成果相関 | 熱量・検知リードタイムとアポ獲得の相関係数 | 有意な正相関 |
| 1アポ単価 | 監視運用コスト ÷ 獲得アポ数 | 既存ソース比で低減 |

検証は [[mochica-appointment-model]] の確信度レポート思想に合わせ、**N数と信頼区間を必ず併記**（小Nでの過大評価を防ぐ）。

## 3. 指標の精査（妥当性・バイアス・ゲーミング）

各指標の落とし穴を明示し、安易な「件数で満足」を排除する。

### 3.1 熱量モデルの構成要素は妥当か
- **NEW=10 の過大計上**：現状 NEW は観測初回に必ず発火し、初サイクルは全社が12.0で並ぶ（実走で確認）。**cold-start期間（最初の数サイクル）はランキング無効**として除外すべき。
- **クエリ網による二重計上**：同一社が複数クエリに跨ると `totalJobs` が累積し JOB_UP が連発する（実走で三田電業が18.0=JOB_UP+NEW_QUERY）。これは「露出の広さ」を熱量に混ぜている。意図的だが、**“求人数の実増”と“我々のクエリ被覆の増”を分離**して計測すべき（後者は鮮度ではない）。
- **重みは現状ヒューリスティック**：NEW=10, NEW_GRAD_YEAR=8 等の値は経験則。階層③の成果データで**再フィットするまで暫定**と明記。

### 3.2 サンプリングバイアス
- 求人ボックスの検索結果順（新着順か関連順か）が不明。関連順だと鮮度シグナルが弱まる。→ 掲載日マーカー併用と、可能なら新着ソートのクエリパラメータ調査で緩和。
- 観測は「我々が投げたクエリ」に閉じる。クエリ集合のバイアスがそのままランキングのバイアスになる（自律増殖が偏りを増幅する危険）。→ クエリ多様性のモニタ（職種×エリアのカバレッジ指標）を運用に追加。

### 3.3 偽シグナル・ゲーミング
- **偽差分**：検索結果のページネーション揺れ・順位変動で同じ社がNEW/GONEを往復しうる。→ **複数サイクルの多数決**（K回連続で消えて初めてGONE確定）と掲載日マーカーで実変化を裏取り。
- **媒体側の演出**：媒体が「新着」表示を出稿テクで操作する可能性。単一マーカー依存を避け、複数シグナル（掲載日＋求人数変化＋卒年）の合議にする。

### 3.4 統計的注意
- 階層③は初期N数が小さい。**2週間サイクル（`KPI_CYCLE_DAYS=14`）で帯別比較**し、N<30の帯は判定保留。リフトは対照群（同期間の従来リスト）と同条件で比較し、季節性（採用解禁時期）を交絡因子として記録。

## 4. 推奨実装（Blocker解消＝優先度最高）

### 4.1 監視専用のキャッシュ制御（Blocker①）
- `run.js` 起動時に監視サイクル長に応じて `process.env.SCRAPE_CACHE_TTL_MS` を間隔の1/2へ設定（例: 60分間隔→30分TTL）。これで同一クエリでも実フェッチが走り差分が成立。
- 中長期：`polite.js` に条件付きGET（If-None-Match / If-Modified-Since、304なら「変化なし」を確定情報として扱う）を追加。帯域とサーバ負荷を抑えつつ変化検知を高精度化。

### 4.2 真の掲載鮮度の取り込み（Blocker②）
- `snapshot.js` のカード抽出に**掲載日パーサ**を追加（「新着」「本日」「N日前」「YYYY/MM/DD」をHTMLから取得済＝実データで確認）。
- 熱量へ反映：観測ベースNEWの重みを下げ、`recencyDays ≤ 3` を強シグナルに昇格。`heat` の icpFactor に鮮度係数（直近ほど高）を追加。
- これにより cold-start でも初回から「実際に新しい掲載」を熱く採点でき、3.1の擬似鮮度問題が解消する。

### 4.3 偽差分の抑制（3.3対応）
- GONE確定をK回連続不在に変更（`diff.js`/`heat.js` に未確定状態を追加）。
- 同一社の複数サイクル観測で求人数の実トレンドを平滑化。

## 5. 改善プロセスの定義（クローズドループ）

設計#9「件数で満足せず下流で評価し、低利回りを止め高利回りに寄せる」を本監視層にも適用する。

### 5.1 評価サイクル（2週間）
`source-kpi.js` に監視CSV（`data/monitor/hottest.csv`、系統C列構成済）と outcomes.csv を投入し、熱量帯別の接続/アポ/商談/受注利回りを2週間毎に算出。

### 5.2 熱量の重み学習ループ
1. 各サイクルの (イベント特徴量, その後のアポ獲得) を蓄積。
2. 2週間毎にロジスティック回帰/単純な勾配調整で `eventWeight`・`icpFactor`・半減期を再フィット。
3. 旧重みと新重みをバックテスト比較（鮮度的中率・リフト）し、改善時のみ採用（チャンピオン/チャレンジャー方式）。

### 5.3 セレクタ較正ループ
`selector-stale` 検知 → （GEMINI_KEYあれば）キャッシュHTMLからセレクタ再導出を試行 → **人手承認ゲート** → adapter反映。媒体DOM変更への自己修復を半自動化。

### 5.4 ガバナンス・ループ（月次）
- クエリ進化の妥当性レビュー（増殖の偏り・暴走バックストップ作動の有無）。
- 媒体追加時の**利用規約・robots再確認**（1.2のリスク管理を定例化）。
- データドリフト監視（観測社数・偽差分率の異常検知）。

### 5.5 ダッシュボード化
`hottest.md` に加え、技術KPI（取得成功率・セレクタ健全性・サイクル時間）と価値KPI（帯別アポ率・リフト）の時系列を1枚に集約し、毎サイクル自動更新。

- 2サイクル目: 大阪クエリ起動で20社新規、東京はキャッシュヒットで差分0（=Blocker①の実証）、クエリ自己増殖+13/+7。
- 求人ボックスHTMLに鮮度マーカー実在を確認: 「新着・本日・2日前・7日前・10日前・20日前」（=Blocker②の好材料）。

## 付録B: 媒体タイプ別「鮮度取得モード」の分類（反復実証で確定）
全静的OK媒体を一律に「掲載日で鮮度判定」できると当初想定したが、実地で2タイプに分かれることが判明（リクナビ企業カードを実HTML検査: 掲載日もエントリー状態も静的HTMLに0件＝較正不足でなく構造的特性）。

| 媒体タイプ | 例 | 静的HTMLの粒度 | 鮮度の取り方 | システムでの扱い |
|---|---|---|---|---|
| 求人アグリゲータ型 | 求人ボックス, (Indeed/スタンバイ) | 求人カード単位 | **掲載日ベース**「N日前/新着」を直接取得 | `recencyDays`→鮮度係数×1.5〜 |
| 新卒ナビ型 | リクナビ, マイナビ | 企業名中心・メタ薄 | **出現ベース** 企業の新規登場=NEW/エントリー開始 | `recencyDays=null`(中立)＋NEW/REAPPEAREDイベント |
| 口コミ・エージェント・逆求人 | みん就/キャリアチケット/OfferBox等 | — | 監視対象外 or presence のみ | 対象外/presence |

含意: 「鮮度」は単一指標でなく媒体タイプで取得経路が異なる。アグリゲータ型＝真の掲載日、新卒ナビ型＝出現検知。
システムは両方を同一の熱量モデルで吸収（掲載日係数 or NEWイベント）し、無い情報を捏造しない（recencyDays=nullは中立）。
新卒ナビ型で掲載日まで取るにはJS描画(Playwright)で詳細ページを開く必要があり、コスト対効果は要検討（出現ベースで実用上は足りる見込み）。

## 付録C: 方法論訂正 — ホームページ判定の偽陽性と検証済エンドポイント
反復実証で、Iteration1の「STATIC_OK+FRESH」判定の一部が**検索URLでなくホームページを叩いたことによる偽陽性**と判明。
LabBase/薬キャリ/レバテックルーキーのホームHTMLを実検査すると、企業ロゴのカルーセル(logoItem×34, companyImageWrapper×21等)や特集セクションであり、掲載日末端要素は0〜2件＝「検索可能な求人/企業一覧」ではない。probeのcompHits/companyLinksヒューリスティックがロゴ壁を企業密度として水増ししていた。

**結論: 「一覧として監視可能」と信頼できるのは検索/一覧URLで叩いた媒体のみ。** ホームページ判定の媒体は実エンドポイント検証が必要。

エンドポイント探索(discover-endpoint.js=トップHTMLリンク解析)で特定した検証済URL:
| 媒体 | 検証済 一覧URL | 企業/鮮度 | 種別 |
|---|---|---|---|
| 求人ボックス | (検索)/新卒…の仕事 | 24/多 | アグリゲータ(掲載日◎) ※観測器実装済 |
| リクナビ | /2027/search/?kw= | 30/出現 | 新卒ナビ(出現ベース) ※観測器実装済 |
| キャリタス就活 | /employment-search/ | 13/鮮度12 | 新卒ナビ(掲載日あり) ※観測器化可 |
| type就活 | /search/event-company | 6/鮮度31 | 新卒ナビ/イベント(掲載日あり) ※観測器化可 |
| ワンキャリア | /companies | 0/link71 | 一覧あるが社名JS描画 |

逆求人/スカウト型(LabBase/レバテックルーキー/OfferBox/キミスカ)とエージェント(キャリアチケット)・口コミ掲示板(みん就)は、公開の求人一覧を持たず**presence/対象外**。
→ 実証の正味: 検索URLを正しく特定すれば新卒ナビ系は掲載日付きで監視可能。専門職の逆求人系は静的監視に不適（公開一覧が無い構造）。
