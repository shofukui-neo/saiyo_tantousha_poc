# 究極の営業リスト作成アプリ — 現状実装の全体像・データフロー・品質アーキテクチャ

> 作成日: 2026-06-18 / 対象: `poc/` 配下の稼働中実装一式
> 目的: 「いま実際に機能している実装」を漏れなく洗い出し、データフローと具体処理を文書化し、**なぜそれが“アポの取れる高品質リスト”に繋がるのか**をアーキテクチャとして説明できるようにする。

---

## 0. 結論（30秒サマリ）

- このシステムは **「企業の母集団を作る層」→「1社ずつデータを肉付けする層」→「複数ソースを名寄せ統合する層」→「アポ取得期待値で採点する層」** の4段で構成される。
- **最終成果物は [`leads-mochica.scored.csv`](../leads-mochica.scored.csv)（1,003社）**。電話番号 **100%**・メール **93.4%**・代表者名 **50.6%** が充足し、各社に **アポ期待度(0-100)・優先度・確信度・「なぜ今・なぜこの企業」** が付与済み。
- 品質の核心は **「件数で満足せず、テレアポでアポが取れる確率を5次元で因果モデル化し、各次元に“実データか代理推定か”の確信度を併走させる」** 点。汎用ICPスコアと決別した [`mochica-fit.js`](../src/mochica-fit.js) がその心臓部。
- **APIキーが無くても全工程が動く**（Bing検索＋正規表現＋MX篩い）。`.env` にキー（gBizINFO/国税庁/Gemini/Hunter）を入れた項目だけ高精度経路へ自動点火する“キー無しで動き、キーがあれば点火”設計。

## 3. 各層の具体処理（実コード根拠）

### ① 母集団ビルダー

母集団の作り方そのものがリスト品質の上限を決める。3本のビルダーは「異なる軸で良い面を確保する」役割分担になっている。

**`build-gbiz.js`（代表者名つき1000）**
- 都道府県コードをラウンドロビンしつつ `gbizSearch` で法人番号を**目標×4.5倍に過剰収集**。
- 各社を `gbizGet` で詳細化（代表者名・公式URL・従業員数・設立年・事業概要）。
- **代表者名が空でない行だけ採用**して目標まで詰める→ 代表者名率 ~96-98%。
- 数件ごとに CSV＋ジャーナル（`data/gbiz-records.json`）をフラッシュ＝**途中再開可能**。gBizは429レート制限があるため過剰収集と再開性が肝。

**`build-fresh-list.js`（上場除外・完全新規1000）**
- 二重フィルタ：`data/listed-bango.json`（EDINET上場法人番号）＋`listed-names.json`（正規化上場名）＋`existing-bango.json`（既存リスト）を、**候補段階と詳細取得後の2回** `isExcluded` で弾く（詳細API無駄打ちを削減しつつ取りこぼしを防ぐ二重の安全網）。
- `normName` で法人格・全角半角・スペースを吸収して名寄せ。過剰収集比率を×10に厚くして除外で痩せる分を吸収。
- → 上場0%・既存重複0%・代表者名100%。

**`build-telapo-1000.js`（電話100%・架電ベース）**
- `data/merged-records.json` を **電話有→URL有→代表者名有**の優先度でソート。
- 各社HPを `polite.js` 経由でクロール（robots遵守）→ **JSON-LD（schema.org）優先**で電話抽出、無ければ正規表現（`phone.js`）。
- **電話が取れた社だけを accepted に追加**＝構造的に電話100%を保証。
- 採用担当者名は採用/問い合わせ系ページに限定して抽出を試行。
- → これが MOCHICA の**架電ベース**。`leads.master.csv` は電話3/1408・担当者0で架電不能だったため、ベースを telapo に切替えた経緯。

### ② 肉付けエンリッチ（媒体実取得層）

**取得は全て [`polite.js`](../src/polite.js) を通る**（これが合法性・安定性の土台）：
- ホスト別**直列化**（同一ホストは前タスクを await してから次、間に `DELAY_MS`=4秒）。
- **robots.txt 遵守**（`isAllowed` がfalseなら `blocked` でスキップ＝ゴミを後段に流さない）。
- **SHA1(url)キーの7日ディスクキャッシュ**（`data/scrape-cache/`、大量の `.json` がこれ）。
- 軽いリトライ（線形バックオフ）。`render=static`（省メモリ）/`auto`（Playwright描画）を選択。

**`scrape-base.js`（媒体実取得の共通基盤）**
- Chromium を `AutomationControlled` 無効化で起動、`humanClick`（boundingBox→mouseをゆっくり6-8step移動→click）/`humanType`（1字ずつ遅延）で**人手のカーソル操作を模倣**しbot検知を回避。
- `networkidle` は常時接続広告で到達しないため `domcontentloaded`＋固定 `SETTLE_MS`(4秒) 待ち。
- 共通抽出器：`extractRecruiterName`（部署→氏名の構造位置＋姓辞書）/`extractIntentSignals`（募集職種・採用予定人数・卒年）/`extractCompanyFacts`（電話・従業員数・資本金・設立）。

**`scrape-mynavi.js`（採用担当者名の最強裏取り）**— 2026-06 実DOM較正済み
1. フリーワード検索GET → `corp{ID}/outline.html` を**社名一致**で特定 → カーソルでクリック。
2. `is.html`/`employment.html` の **『問合せ先』ブロック**から `部署＋氏名＋電話＋メール` を構造抽出（例: `問合せ先 人事総務部 野崎瑠美 03-… saiyo@…` で全項目取得を実証）。
3. **GEO_SUFFIX ガード**（関東支店・宮城県 等の地名を弾く）＋「部署直後」という構造位置で辞書外の姓を許容（精度優先＝誤った担当者名はゼロ件より悪い）。
> 含意：マイナビの問合せ先は中堅大手でも個人名が載ることがあり、「自社サイトでは個人名非公開」という母集団の壁を**媒体側の別経路で突破**する手段。

**`build-media.js`/`scrape-media.js`（媒体掲載の濃縮）**
- 求人ボックス/Indeed から「今採用中」を検知（`E-hiring.csv`）。
- マイナビ/リクナビ/ワンキャリアの掲載有無を判定（`A-media.csv`）。正規化社名で一致確認し誤マッチを防止。

### ③ 名寄せ統合（多系統マージ→4次元採点）

**`sources/manifest.json` が系統を宣言**：A=採用メディア起点（新卒フラグ）/ B=企業属性（gBiz）/ C=intentトリガー（PR TIMES・求人検索）/ D=ネットワーク（失注先）。各ソースに `system`・`source`（KPI帰属単位）・`intent`（既定★）・`cost_per_company` を持たせる。

**`merge.js`（役割固定マージ）**
- **突合キー＝法人番号（無ければ正規化社名：法人格・㈱・全角・記号を吸収）** で重複排除。
- **役割固定**（`FIELD_OWNERS`）：新卒シグナルは系統A、属性は系統B、トリガーは系統Cの値を優先採用。連絡先（採用担当者名/メール/電話/役職/部署）は**確度列を見て最高確度を採用**。
- 各行に `取得元媒体/系統/起点系統/起点ソース/ソース数/新卒フラグ/intent★/トリガー/名寄せキー` を刻む（トレーサビリティとKPI帰属の土台）。
- **新卒フラグ** = 系統A存在 OR `SHINSOTSU_FLAG_COLS` truthy。**intent★(0-5)** = (系統既定intent + manifest intent) + トリガー列truthy数、上限5。

**`quality.js`（4次元加重スコア・0-100）**
```
総合 = 0.30×ICP適合 + 0.35×採用インテント + 0.20×データ品質 + 0.15×タイミング − ペナルティ
```
- **① ICP適合(30%)**：従業員規模(40%, スイート100-250=100/レンジ内70/外25) ＋ 業種一致(30%) ＋ 地域一致(15%) ＋ 継続信用(15%, 設立10年+30/補助金+20)。
- **② 採用インテント(35%・最重要)**：**新卒フラグ/出稿データがあれば実シグナル採点**（shinsotsu40%＋媒体数25%＋継続20%＋金額15%、＋系統Cトリガー★で最大+30）。無ければ**代理推定**（基本20＋担当者名+40＋採用ページ+25、`インテント根拠=代理推定`と明示）。
- **③ データ品質(20%)**：電話妥当(30%)＋メール確度(25%)＋担当者/部署(20%)＋URL(15%)＋鮮度(10%)。
- **④ タイミング(15%)**：新卒採用月係数（9-12月計画ピーク100/1-3月85/4-6月70/7-8月55）＋**ゴールデンタイム**（辞退シグナルで満点・来期検討で加点）。
- **属性ランク** A(ICP≥75)/B(≥50)/C。**優先度割り込み**：intent★≥3 かつ 属性≠C なら中位スコアでも「今週架電」へ1段昇格（“今動いている”を逃さない／属性不明は割り込ませず先に系統Bで肉付け）。
- **ペナルティ**：除外フラグ/DNC −100、既存顧客 −60、競合ATS −30。

### ④ MOCHICA向け再採点（最終成果・心臓部）

汎用ICP（quality.js）と**決別**し、「新卒を増やしたい中小(50-150名) × 28卒の媒体選定・採用設計期 × 到達性（電話＋担当者名）」に特化した [`mochica-fit.js`](../src/mochica-fit.js)。

```
アポ期待度 = Σ(次元スコア × 重み) − ペナルティ
```

| 次元 | 重み | 階級化ロジック（実在性で確信度が変わる） |
|---|---|---|
| **① 新卒インテント** | **0.34** | マイナビ/リクナビ実取得=100/95 ＞ 新卒フラグ確定=88 ＞ 求人検索で採用中=72 ＞ 採用ページのみ(代理)=40 ＞ 無=10。複数媒体出稿+8/募集3職種+5/28卒明示+6 |
| **② 規模フィット** | **0.26** | 50-150名=100、30-50=72、150-250=76、250-500=40、500-1000=20、**1000+=8（大企業＝自前/競合ATS濃厚で強減点）**、<20=25 |
| **③ 到達性** | **0.20** | 電話妥当+60、採用担当者名+30（名指し架電可）、代表者名のみ+8、部署/役職+10（max100） |
| **④ タイミング** | **0.14** | 28卒設計期月係数（6-8月=サマーインターン準備＋媒体選定の山92-95、秋冬ピーク90-100）＋辞退/更新トリガーでレコード単位に引上げ |
| **⑤ 継続・信用** | **0.06** | 設立年（毎年新卒を採れる体力）＋補助金 |

- **ペナルティ**：競合ATS −45 / 大企業1000+ −20 / 既存顧客 −70 / DNC −100。
- **確信度(confidence)**：各次元の確信度を「スコア×重み」で加重平均。マイナビ実取得95%・新卒フラグ80%・代理推定35% を集約し、**スコアが実データ由来か代理推定由来かを可視化**。
- **「なぜ今・なぜこの企業」(why)**：INT/SIZE/REACH/TIM の根拠を `｜` で連結したワンライナー（営業がそのまま読める）。
- **確信度レポート**（コンソール）：上位N社の `verifiedIntent(INT確信度≥80) / sizeFit(SIZE≥90) / callable(電話妥当) / named(担当者名あり)` の割合＝**“今のリストよりアポが取れる”と言い切るための裏取り**。

**`build-mochica-list.js`（駆動）**
- base＝telapo（電話100%）に daihyou/master/A-pages を法人番号→社名で合流（連絡先＋新卒インテントを補完）。
- `--enrich N`：上位N社をマイナビ実取得→採用担当者名を補完→**再採点**（INTとREACHが上がるため）。
- 出力先頭に `アポ期待度/優先度/確信度/なぜ今なぜこの企業/INT/SIZE/REACH/TIM/TRUST/採点根拠` を付与。

### ⑤ KPIループ（`source-kpi.js`）

- `outcomes.csv`（法人番号 or 企業名キー＋接続/アポ/商談/受注/コスト）をリードに突合。
- ソース別に 適合率・接続率・接続→アポ率・商談化率・受注率・1社コスト・アポ単価/受注単価 を算出。
- **利回り中央値×1.2以上＝「寄せる」／×0.6以下＝「止める検討」**（母数<5は判定保留）を自動提示。2週間サイクルで配分最適化。

## 5. なぜこれが「品質の良いリスト」に繋がるのか（アーキテクチャの論理）

品質を生む設計判断を7点に分解する。これがこのシステムの“思想”である。

### 原則1：品質＝件数ではなく「アポが取れる確率」で定義した
汎用的な「ICP適合度」採点をやめ、`mochica-fit.js` で **テレアポでアポが取れる因果（規模×新卒インテント×電話×担当者名×タイミング）** を直接モデル化した。リストの良し悪しを「下流の受注率/アポ率」で測る `source-kpi.js` まで含め、**指標がビジネス成果に接続**している。

### 原則2：母集団の作り方で上限を決める（3軸の面取り）
代表者名つき(daihyou)・完全新規(fresh)・電話100%(telapo) の3母集団を別軸で確保。**「電話が取れた社だけ採用」「代表者名がある社だけ採用」「上場・既存を二重に除外」** という“入口で品質を作り込む”設計により、後段の採点が砂上の楼閣にならない。

### 原則3：実データと代理推定を区別し、確信度を併走させる
最重要次元（新卒インテント34%）を **マイナビ実取得＞新卒フラグ＞求人検索HIT＞採用ページのみ** と階級化し、各社のスコアが「何由来か」を `確信度` と `インテント根拠` で常に可視化。**“言い切れる根拠”を持つリード**が上位に来る。確信度レポートは営業が顧客の前で根拠を言える状態を作る。

### 原則4：名寄せ（法人番号）が統合の品質を担保する
複数ソースを**法人番号優先・正規化社名フォールバック**で1社1レコードに畳み、**役割固定マージ**で「新卒シグナルはA、属性はB、トリガーはC」と出所の信頼度に応じて値を採用。重複や属性の取り違えを構造的に抑止する。

### 原則5：“今動いている”を取りこぼさない優先度割り込み
intent★≥3（PR TIMES・求人増・採用ページ更新等のトリガー数）かつ属性が判明している行は、総合スコアが中位でも**「今週架電」へ昇格**。辞退シグナルでタイミング満点。**鮮度の高い購買シグナルを静的スコアより優先**する設計。

### 原則6：合法・安全な取得が継続供給を可能にする
全取得が `polite.js`（robots遵守・ホスト別レート制限・7日キャッシュ・人手模倣カーソル）を通る。**公開ページのみ・認証回避なし**で、止まらず回り続けられる。スクレイピングが規約に触れうる系統A/Cは「許諾済みCSVの取り込み」に徹し、自動化は公開API（gBiz）に寄せる。

### 原則7：キー無しで動き、キーで点火する（縮退しても止まらない）
gBiz/国税庁/Gemini/Hunter のキーが無くても Bing検索＋正規表現＋MX篩いで全工程が通る。キーを入れた項目だけ高精度経路に**自動で切り替わる**ため、運用は段階的に品質を上げられ、どこかが落ちても全体は止まらない。

## 6. 既知の限界と次の一手

| 限界 | 真因 | 対策の方向 |
|---|---|---|
| **採用担当者名 0%** | 抽出技術でなく**母集団問題**：中堅〜大手は自社サイト/メールに個人名を出さない。Wantedlyも対象社を返さない（`company-mismatch`）。 | ①母集団をSME/ベンチャーへ寄せる ②マイナビ「問合せ先」実取得（`mochica:enrich`）を主経路化 |
| `A-pages.csv`/`A-names.csv` 未生成 | 媒体横断/Wantedly層はコードのみで実走未完 | `npm run pages`/`npm run names` を母集団選定後に実行 |
| C-prtimes/D-shitchu がサンプル | intent/ネットワーク系は許諾データ前提で未投入 | HRog出稿検知・PR TIMES・失注先CSVを定期供給 |
| outcomes が7件 | 架電実績の書き戻しが未蓄積 | 架電→`outcomes.csv`→`source-kpi.js` の2週間ループを起動 |
| Gemini/gBizトークン待ち | 採用担当者AI抽出・構造化発掘の高精度経路が未点火 | `.env` にキー投入で自動点火（コード変更不要） |
