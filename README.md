# 企業情報 自動取得 PoC

**①企業名そのものの自動収集（発見層）→ ②公式HPのURL → ③電話番号 → ④採用担当者名 を自動取得する**実装です。
企業名リストを持っていれば②〜④だけを回せますし、企業名すら無ければ**キーワード（業種・地域）や一覧ページから企業名自体を自動収集**できます。
入出力は **Google スプレッドシート / GAS / ローカルCSV**。スクレイピング本体はGASでは動かせないため Node＋Playwright で実行し、**リストの読み込みと結果の書き戻し**をスプレッドシートが担います。

> **🆕 究極の営業リスト作成アプリ（一気通貫・統合版）→ 末尾の [統合アプリ](#統合アプリ究極の営業リスト作成アプリ一気通貫) を参照。**
> 旧GASの6レイヤ（ICP生成→企業発掘→法人番号名寄せ→採用担当者→メール→Tier）をローカルへ移植し、本PoCの強み（Playwright描画・電話番号抽出・URL検証）と合体。`企業選定→各種データ収集→担当者マスタ出力` を **1コマンド**（`node src/app.js`）で実行します。**APIキーが無くても動き**、キー（gBizINFO/国税庁/Gemini/Hunter）を入れた項目だけ自動で高精度経路に点火します。

> **🎯 MOCHICAアポ取得期待値モデル（リスト再設計）→ [MOCHICA向けリスト](#mochica向けリストアポが取れる基準で選定) を参照。**
> 「完成リストが今のリストよりアポを取れる」と言い切るための再設計。汎用ICP採点と決別し、**新卒を増やしたい中小(50-150名) × 28卒の媒体選定・採用設計期 × 到達性（電話＋採用担当者名）**で採点する `mochica-fit` を新設。`npm run mochica` で採点＋「なぜ今・なぜこの企業」付きCSV＋**確信度レポート**（上位の何割が新卒採用を実データで確認済み／架電可能か）を出力。担当者名は `npm run mynavi`（Playwrightカーソル操作・マイナビ実取得）で補完する。

## 起点は3通り（どれでもOK）
1. **企業名すら無い** → `--discover "<キーワード>"`（業種・地域など）または `--discover-url <一覧ページURL>` で企業名を自動収集 → そのまま②〜④へ。
2. **企業名リストはある** → A列に企業名（URLは任意）。URLが空なら企業名から自動発見。
3. **URLも分かっている** → A列＋B列に記入。検索をスキップして高速・確実。

## 設計方針（外部AI API 不使用・ローカル処理のみ）
**Anthropic 等の外部AI APIは一切呼び出しません。** 課金もネットワーク先での処理も発生しません。全項目をローカルの正規表現・スコアリング・人名判定で取得します。
- **URL発見**：Bing の検索結果HTML（**APIキー不要・無料**）から候補を取得 → 求人媒体・SNS・企業DBを除外 → 「公式コーポレートサイトらしさ」をスコアリング → 上位を実際に取得して**企業名一致を検証**してから採用。
- **電話番号**：`tel:` リンク＋正規表現で抽出。`TEL/代表/お問い合わせ` 近接で加点、`FAX` 近接は減点、法人番号・登録番号など長い数字列の誤検出を除外。
- **採用担当者名**：本文の正規表現（「採用担当：氏名」等）＋**人名らしさ判定**（役割語・組織語・見出し語を除外）で抽出。

## なぜ媒体が「公式企業サイト」なのか
- **合法性が最も安全**：企業自身の公開ページを robots.txt 遵守・低速で読むだけ。第三者プラットフォームのアンチボットや認証回避を伴わない。
- **信号が強い**：電話番号・採用/人事担当の氏名が「お問い合わせ」「採用情報」「会社概要」に載りやすい。
- **技術的に扱いやすい**：多くは静的HTMLで取得でき、必要時のみブラウザ描画にエスカレーションすればよい。

## スプレッドシートの列レイアウト
1行目はヘッダ。**A列（企業名）を用意するだけ**で動きます。B列（URL）は任意で、空なら企業名から自動発見します。C列以降は本ツールが自動で書き戻します。

| 列 | 内容 | 記入者 |
|---|---|---|
| A | company_name（企業名・**必須**） | あなた |
| B | homepage_url（公開トップページURL・**任意**） | あなた（空欄なら自動発見。分かっていれば入れると高速・確実） |
| C〜R | status / resolved_url / phone / name / role / department / confidence / url_source / phone_source_url / name_source_url / evidence / engine / pages_checked / elapsed_ms / error / updated_at | ツールが書き戻し |

- `status` は **採用担当者名** の HIT / MISS（加えて NO_URL / SKIP_ROBOTS / ERROR）。URL・電話の取得可否は `resolved_url` / `phone` 列の有無で分かります。
- `url_source` は URL の出所：`input`（入力）/ `search+verified`（検索＋企業名一致を確認）/ `search(unverified)`（検索のみ・要目視確認）。
- 再実行時に `ONLY_PENDING=true` にすると **status空欄の行だけ**処理します（差分更新・再開に便利）。

## 接続方式は2つ（どちらか選択）

### 方式①：Google Sheets API（サービスアカウント）— 自動実行向き・推奨
1. GCPでプロジェクト作成 → **Google Sheets API を有効化**
2. **サービスアカウント**を作成 → 鍵（JSON）を作成・ダウンロード（例：`service-account.json` をプロジェクト直下に置く）
3. 対象スプレッドシートを、サービスアカウントのメール（`xxxx@xxxx.iam.gserviceaccount.com`）に **「編集者」で共有**
4. `.env` に設定：
   ```
   SOURCE=sheet
   SHEET_ID=<スプレッドシートID>      # URLの /d/ と /edit の間
   SHEET_TAB=Sheet1
   GOOGLE_APPLICATION_CREDENTIALS=./service-account.json
   ```

### 方式②：GASウェブアプリ橋渡し — GCP不要（GAS運用に慣れている場合）
1. 対象スプレッドシートを開く → **拡張機能 → Apps Script**
2. `gas-bridge/Code.gs` を貼り付け（必要なら `TAB` をシート名に変更）
3. **デプロイ → 新しいデプロイ → ウェブアプリ**（実行＝自分／アクセス＝全員）
4. 発行された `/exec` のURLを `.env` に設定：
   ```
   SOURCE=gas
   GAS_URL=https://script.google.com/macros/s/XXXXX/exec
   ```

### 発見モード（企業名そのものを自動収集）
```bash
# キーワード（業種・地域など）から企業名を収集し、そのままURL/電話/担当者まで取得（結果はCSV）
node src/index.js --discover "渋谷 システム開発 会社" --limit 20

# 企業一覧ページ（商工会名簿・会員一覧・まとめ記事など）から企業名を収集
node src/index.js --discover-url "https://example.org/members" --limit 50

# 企業名リストだけ作る（C列以降は取得しない）。出力CSVをそのまま --input に使える
node src/index.js --discover "大阪 製造業 企業 一覧" --discover-only --out companies.csv
```
- 発見モードの結果は**既存シートを上書きしないようCSVに出力**します（`--out` で変更可）。
- **収量はソース依存**：多くのWebページは「株式会社」を省いて製品名・ブランド名で記載するため、`おすすめ◯◯` 系の記事は法人名が少なく収量が落ちます。**会社名が法人格付きで列挙される一覧（会員名簿・企業ディレクトリ・「◯◯ 会社 一覧」）ほど高収量・高精度**です。精度優先のため、確信が持てない断片は採用しません（ゴミを後段に流さない）。

## 出力（4項目）
- **企業名 → 公式URL → 電話番号 → 採用担当者名** を各行 C〜R列に直接書き戻し（出所・根拠・取得日時付き）
- コンソールに **公式URL発見率 / 電話番号取得率 / 採用担当者名取得率**、平均確度・処理時間を集計表示
- 書き戻しに失敗した場合は `results.fallback.csv` に退避（結果を失わない）

→ セグメント（業種・規模・ベンチャー比率）ごとにシート／タブを分けて回すと、**セグメント別の各取得率**がそのまま取れます。

## 処理の流れ
0. **発見層（任意）** `src/discover.js`：キーワード or 一覧URLから企業名を自動収集（Bing検索結果＋上位ページの構造要素から法人格付き社名を抽出）。
1. **URL発見** `src/search.js`：入力URLがあれば使用。無ければ企業名でBing検索 → 候補スコアリング → 上位を取得して企業名一致を検証。
2. **巡回** `src/fetch.js`：トップを取得し、ナビから会社概要/お問い合わせ/採用ページを発見。リンクが無ければ定番パス（/company, /contact, /recruit …）と corp. サブドメインを推測補完。
3. **抽出** `src/phone.js`（電話・正規表現）＋ `src/extract.js`（担当者名・正規表現＋人名判定。**外部AI API不使用**）。
4. **検証ゲート** `src/validate.js`：担当者名は人名らしさ＋採用/人事ロール＋確度閾値を通過したものだけHIT。
5. **集計・書き戻し** `src/metrics.js` / `src/io-common.js` ＋ 各I/Oアダプタ。

## 設計との対応
| 要素 | 本PoCの実装 |

| URL発見（企業名→公式HP・APIキー不要） | `src/search.js` Bing検索HTML → 除外/スコアリング → 企業名一致を検証 |

| 入出力＝スプレッドシート | `src/sheets.js`（Sheets API）/ `src/gas.js`＋`gas-bridge/Code.gs`（GAS橋渡し）/ `src/io-common.js`（共通ロジック） |
| RPA（動的ページ対応） | `src/fetch.js` 静的取得→`looksJsRendered`→Playwright描画にエスカレーション。ページ発見＋定番パス推測 |
| 担当者名抽出（外部AI API不使用） | `src/extract.js` 本文の正規表現＋人名らしさ判定（役割語・組織語・見出し語を除外） |
| 検証ゲート（担当者名HIT判定） | `src/validate.js` 人名らしさ＋採用/人事ロール＋確度閾値 |

## 法令・マナー（重要）
- robots.txt と各サイト規約を尊重し、**公開ページのみ**を対象にする（認証回避はしない）。
- 取得した担当者名は**個人情報**。利用目的の特定・適正取得・オプトアウト対応を前提に運用すること。
- `.env` の `USER_AGENT` を連絡先付きに設定し、`POLITE_DELAY_MS`/`CONCURRENCY` を過負荷にならない値で。
- 本コードは検証用であり、**法的助言ではありません**。運用前に規約・関連法令の最終確認を。

## 精度の限界（重要・既知）
- **発見層の収量はソース依存**：法人格（株式会社等）付きで社名が列挙されるページほど高収量。製品名・ブランド名だけの記事は収量が落ちます。確実な断片のみ採用するため、ひらがな始まりの一部社名などは取りこぼすことがあります（精度優先）。
- **同名企業の取り違え**：企業名だけでは、よく似た社名の別企業の公式サイトを掴むことがあります。`url_source` が `search(unverified)` の行や、規模に対して不自然なドメインは**目視確認**を推奨。確実な行はB列にURLを入れて再実行してください。
- **担当者名の精度**：氏名抽出はローカルの正規表現＋人名判定で行うため、定型的な表記（「採用担当：氏名」等）は拾えますが、文章中に埋もれた氏名や独特なレイアウトは取り逃すことがあります（**外部AI APIは使いません**）。
- **大企業のコーポレート分離**：採用/会社情報が別サブドメイン（例 corp.example.co.jp）にある場合に対応済みですが、巡回ページ数 `MAX_PAGES_PER_SITE` 内で見つからないと電話を取り逃すことがあります。

## チューニング
- `SEARCH_ENGINE`：`bing`（既定）/ `duckduckgo`。環境により片方が弾かれることがあるので切替可能。
- `MAX_PAGES_PER_SITE`：巡回ページ数。増やすと電話/担当者の取得率↑・所要時間↑。
- `CONFIDENCE_THRESHOLD`：上げると担当者名の精度↑/取りこぼし↑。
- 担当者名の語彙（役割語・非人名ブロックリスト）は `src/extract.js` / `src/config.js` で調整可能。

## 次のステップ（Phase 1）
1. 発見層の高度化：gBizINFO（法人番号・無料トークン）など構造化ソースと併用し、業種・地域での網羅的な企業名収集と収量・精度の両立。
2. 同名企業対策：所在地・業種などの補助シグナルで候補を絞り込み、取り違えを低減。
3. 媒体ワーカーを追加（PR TIMES / Wantedly 等）し、**並列レース＋first-validated-hit cancel** 化。
4. 全MISS時の**架電確認→スプレッドシート書き戻し→次サイクル昇格**ループを実装。

# 統合アプリ「究極の営業リスト作成アプリ」（一気通貫）

旧GAS（採用リスト作成システム REV.3）の6レイヤをローカルへ移植し、本PoCの強み（Playwright描画・電話番号抽出・URL一致検証・robots遵守）と合体させた**一気通貫パイプライン**です。エントリは [`src/app.js`](src/app.js)。

```
[L1 ICP ]  商材→理想顧客像   : Gemini(あれば) / 手動設定(.env)        src/icp.js
[L2 発掘]  企業選定          : gBizINFO(あれば) / Bing発掘(既存)        src/discovery.js
[L2 名寄]  法人番号/代表者   : 国税庁 + gBizINFO(あれば・任意)          src/nta.js, src/gbiz.js
[L3 担当]  採用担当者レース  : Gemini(あれば) / 正規表現(既存)          src/recruiter.js（+ fetch/extract/validate）
[L3 電話]  電話番号          : 既存 src/phone.js（GASには無かった）
[L4 mail]  メール            : MX篩い+パターン / Hunter(あれば・任意)    src/email.js
[L5 出力]  Tier/架電呼称/出力 : 担当者マスタ(Sheet upsert + CSV)         src/score.js, src/master-io.js
           司令塔            :                                          src/pipeline.js, src/app.js
```

## 思想：キー無しでも動き、キーがあれば点火する
**現状APIキーがゼロでも**、`discoverUrl（Bing）＋正規表現抽出＋電話抽出＋MX篩い` だけで企業選定〜担当者マスタ出力まで通ります。`.env` に下記キーを入れた項目だけ、自動で高精度経路に切り替わります（コード変更不要）。

| 環境変数 | 入れると点火する経路 | 無いときのフォールバック |
|---|---|---|
| `GBIZ_TOKEN` | gBizINFO で 業種×地域の**構造化発掘**＋従業員数選別・代表者名/HP取得 | Bing検索ベースの発掘（`discover.js`） |
| `NTA_APP_ID` | 国税庁APIで**商号→法人番号**の名寄せ | 名寄せスキップ（法人番号空欄） |
| `GEMINI_KEY` | **ICP自動生成**＋**採用担当者AI抽出** | 手動ICP（.env）＋正規表現抽出 |
| `HUNTER_KEY`＋`DO_EMAIL_VERIFY=true` | メール実在検証 | MX篩い＋パターン推測（確度0.4） |
| `SHEET_ID`＋サービスアカウント | 「担当者マスタ」タブへ upsert 出力 | ローカルCSV（`leads.csv`）のみ |

## セットアップ
```bash
npm install
npx playwright install chromium      # 動的ページ対応（任意・静的のみなら省略可）
cp ".env copy" .env                  # 統合アプリ用の設定は .env.example の末尾セクション参照
```
発掘の起点になる ICP を `.env` に設定（キー無しの最低限）：
```
ICP_INDUSTRIES=システム開発,SaaS       # 発掘の業種（カンマ区切り）
ICP_PREFECTURES=東京都,大阪府          # 地域（空なら全国）
ICP_EMP_MIN=50
ICP_EMP_MAX=300
DISCOVER_TARGET=100                    # 目標社数
```

## 実行
```bash
# ① ICP(.env) から自動発掘して一気通貫（企業選定→全データ収集→担当者マスタ出力）
node src/app.js

# ② キーワード起点で発掘して一気通貫
node src/app.js --discover "渋谷 システム開発 会社" --limit 30

# ③ 企業一覧ページ起点

# ④ 既に企業名リストがある場合（company_name 列のCSV。homepage_url 列は任意）

# ⑤ 発掘した企業名だけ確認（収集の確認用・データ収集はしない）

# 主なオプション
node src/app.js --limit 50 --concurrency 3 --out leads.csv
```

## 出力（担当者マスタ）
各社1行で `leads.csv`（および設定があればスプレッドシートの「担当者マスタ」タブ）へ出力します。列：

`企業名 / 法人番号 / 採用担当者名 / 役職 / 部署 / 代表者名 / メール / メール確度 / 担当者確度 / 電話番号 / 公式URL / Tier / 取得元媒体 / 根拠URL / 架電呼称 / 業種 / 都道府県 / 従業員数 / 取得日`

- **Tier**：A（担当者＆メール高確度）/ B（どちらか高確度）/ C（代表者名フォールバック可）/ D（要架電確認）。
- **架電呼称**：担当者名が取れなくても必ず生成（例「人事部 新卒採用ご担当者様」）。ICPの部署・新卒/中途シグナルから自動調整。
- スプレッドシート出力は**法人番号→企業名の順でupsert**（重複行を作らない）。`SHEET_ID` 未設定ならCSVのみ。

# リスト品質スコアリング（テレアポ・アポ率の最適化）

作成した担当者マスタ（`leads.csv`）を、テレアポのアポ率を左右する**4ディメンション加重モデル**で採点し、架電優先度を振り分けます。実装は [`src/quality.js`](src/quality.js)（純ロジック）＋ CLI [`src/score-list.js`](src/score-list.js)。

```powershell
node src/score-list.js                 # leads.csv を採点 → leads.scored.csv
npm run score                          # 同上
node src/score-list.js --input leads.csv --out scored.csv --top 30
node src/score-list.js --min 70        # 「今週架電」(総合70+)だけをCSV出力
```

## スコア設計（合計100点・加重平均）
| ディメンション | 既定ウェイト | 見るデータ |
|---|---|---|
| ① ICP適合度 | **30%** | 従業員規模（スイート100〜250名）・業種一致・地域一致・設立継続性・補助金 |
| ② 採用インテント | **35%（最重要）** | 求人出稿シグナル（あれば）／無ければ採用ページ・担当者HITの**代理推定** |
| ③ データ到達性・品質 | **20%** | 電話番号の妥当性・メール確度・担当者/部署・公式URL・データ鮮度 |
| ④ タイミング（季節） | **15%** | 新卒採用スケジュール連動の月係数（前年9-12月=計画ピーク100、1-3月=85、4-6月=70、7-8月=55） |

`総合 = 0.30×ICP + 0.35×インテント + 0.20×データ + 0.15×タイミング − ネガ調整`。
ネガ調整は入力CSVに `除外フラグ`／`DNC`／`既存顧客`／`競合ATS導入` 列があれば自動減点。
ウェイト・閾値は `.env`（`QUALITY_W_*`／`QUALITY_PRIORITY_HIGH`／`QUALITY_PRIORITY_MID`）で調整可能。

## 出力と優先度
採点済みCSVは元の列に `品質スコア / 優先度 / ICP適合 / 採用インテント / データ品質 / タイミング / インテント根拠 / スコア根拠` を追記。総合スコアで降順ソートし、コンソールに各ディメンション平均・優先度内訳・上位社を表示します。
- **今週架電**（総合70+）／**ナーチャリング**（45-69）／**後回し**（<45）の3段階。

## 採用インテントを「実データ」化する（最重要ディメンション）
キー無しの現状では、採用インテント（35%）は**採用ページ有無・担当者HITの代理推定**で採点され、`インテント根拠=代理推定` と表示されます。これを実データに上げるには、入力CSVに求人出稿データの列を足すだけで自動的に本スコアへ切り替わります（HRogリスト等のCSVを社名で結合）：

`新卒出稿`（○/掲載中）・`出稿媒体数`（数値）・`予想出稿金額`（数値）・`出稿継続性`（○）・`現在求人掲載中`（○）

> 求人出稿データは購買意図を60-90日先行して示す最良の早期シグナル。MOCHICA（新卒・アルバイト）用途では、全雇用形態をカバーする HRogリスト等の連携が最も効果的です。

# 多系統ソース統合でリスト精度を上げる（収集戦略の実装）

「**新卒採用フラグ**を軸に逆算し、**件数でなく下流の接続→アポ率**でソースを評価する」という収集戦略を、3つの仕組みで実装しました。1社1レコードのスコアリング（上のセクション）の**手前**に、**複数ソースの統合・名寄せ**を、**後ろ**に**ソース別の利回り評価ループ**を足した形です。

```
[複数ソースCSV] ──merge.js──▶ [名寄せ済みマスタ] ──quality.js──▶ [採点・優先度付きリスト] ──source-kpi.js──▶ [ソース別利回り→配分最適化]
   A 採用メディア(新卒フラグ)      法人番号で重複排除          4次元＋新卒フラグ軸＋★         接続→アポ→受注 でソースを採点
   B 企業属性(gBiz等)            役割固定でフィールド採用      属性ランクA/B/C×intent★        低利回りを止め高利回りに寄せる
   C intent/トリガー             出所(取得元媒体)を刻む        系統C★≥3で優先度を割り込み
   D ネットワーク/既存資産
```

## 系統（ソースの役割）
| 系統 | 何が取れるか | 役割 | サンプル |
|---|---|---|---|
| **A** 採用メディア起点 | **新卒フラグ**＋採用規模＋痛みシグナル | 起点（最優先） | `sources/A-mynavi.csv` |
| **B** 企業属性起点 | 従業員数・業種・地域・設立・補助金・代表者名 | 母集合・名寄せ・肉付け | `sources/B-gbiz.csv` |
| **C** インテント/トリガー | プレスリリース・出稿増・採用ページ更新・辞退・来期検討 | "今動いている"で優先度を上書き | `sources/C-prtimes.csv` |
| **D** ネットワーク/既存資産 | 失注先・既存リレーション・反応者（連絡先付き） | 転換率最大の温かいリード | `sources/D-shitchu.csv` |

## ① 統合ビルド（マージ→採点→出力）
```powershell
node src/build-list.js --sources sources/manifest.json --out leads.master.csv
npm run build-list                                   # 同上（既定manifest）
node src/build-list.js --sources sources/manifest.json --kpi sources/outcomes.csv   # KPIまで一気に
node src/build-list.js --dir sources                 # manifest無し：ファイル名 A-/B-/C-/D- で系統推定
```
`sources/manifest.json` に各ソースを宣言します（`system`=系統 / `source`=KPI帰属の単位名 / `intent`=既定★ / `cost_per_company`=取得コスト）。実行すると:
- **突合キー=法人番号**（無ければ**正規化社名**：法人格・㈱・全角・記号を吸収）で**重複排除**。
- **役割固定マージ**：新卒シグナルは系統A、属性は系統B、トリガーは系統C の値を優先採用。連絡先は確度の高い方。
- 各行に **`取得元媒体`（出所の連結）/`系統`/`起点ソース`/`新卒フラグ`/`intent★`/`ソース数`** を刻む（KPI帰属とトレーサビリティの土台）。
- そのまま4次元採点し、**優先度降順のマスタCSV**と、名寄せ統計・新卒フラグ率・★分布・上位社を表示。

## ② スコアリングの強化（精度の核）
`src/quality.js` を収集戦略に合わせて拡張しました（既存挙動は後方互換）。
- **新卒フラグ軸**：`新卒フラグ` が立っていれば、出稿データが無くても**採用インテントを実シグナル**として採点（`インテント根拠=実シグナル`）。設計の背骨。
- **intent★トリガー（系統C）**：`プレスリリース`/`出稿増`/`採用ページ更新`/`インテント`/`辞退シグナル` の数＝★（0-5）。intentに最大+30点上乗せ。
- **属性ランク A/B/C**：ICP適合スコアで階級分け（A≥75 / B≥50 / C）。`属性ランク × intent★` で優先度を読む。
- **優先度の割り込み**：系統Cの強トリガー（**★≥3**）かつ属性がC級でない行は、総合スコアが中位でも**今週架電へ1段昇格**（"今動いている"を逃さない）。属性不明（C級）は割り込ませず、まず系統Bで肉付けしてから昇格させる設計。
- **ゴールデンタイム**（設計#7）：`辞退シグナル`=○ で**タイミング満点**（夏〜秋の辞退期は最も刺さる）、`来期検討`=○ で加点（秋〜冬の採用設計期）。

## ③ ソース別KPI評価ループ（設計#9「必ず回す」）
件数で満足せず、**下流（接続→アポ→商談→受注）と1社あたりコスト**でソースを採点し、低利回りを止め高利回りに寄せます。
```powershell
node src/source-kpi.js --leads leads.master.csv --outcomes sources/outcomes.csv
npm run kpi
node src/source-kpi.js --leads leads.master.csv --attr touch    # 接触した全ソースに帰属（既定は起点ソース）
node src/source-kpi.js --leads leads.master.csv                  # 成果CSV無し＝カバレッジ(件数/適合率)のみ
```
- **成果CSV**（`sources/outcomes.csv`）は `法人番号`(無ければ`企業名`) をキーに `接続/アポ/商談/受注/コスト` 列。テレアポ結果をここに書き戻して回します。
- ソース別に **適合率・接続率・接続→アポ率・商談化率・受注率・1社コスト・アポ単価/受注単価** を算出。
- 利回り（受注率、無ければアポ率）の**中央値×1.2以上＝「寄せる」／×0.6以下＝「止める検討」**を自動提示（母数<5は判定保留）。**2週間サイクル**（`KPI_CYCLE_DAYS`）で回す前提。

## 運用の型（フェーズ）
1. **濃いセルを作る**：系統A（マイナビ等の新卒フラグ）を起点に、系統B（gBiz）で属性を肉付け、系統C（PR TIMES等）で"今動いている"を割り込ませ、最濃度セル（新卒フラグ×従業員スイート×Aランク業種×痛みシグナル）を**今週架電**へ。
2. **回す**：架電結果を `outcomes.csv` に記録し `source-kpi.js` で**ソース別利回り**を2週間ごとに判定 → 配分を最適化。
3. **継続供給**：HRog出稿検知・プレスリリース・インテントを定期で系統A/Cに追加（CSVを増やしてmanifestに足すだけ）。

## 設定（`.env` / `src/config.js`）
`SYSTEM_INTENT`（系統別★）・`FIELD_OWNERS`（役割固定）・`INTENT_TRIGGER_COLS`（トリガー列）・`SHINSOTSU_FLAG_COLS`（新卒フラグ列）・`GRADE_A_MIN`/`GRADE_B_MIN`（属性ランク閾値）・`INTENT_OVERRIDE_STARS`（割り込み★）・`KPI_FIT_SCORE_MIN`/`KPI_OUTCOME_COLS`/`KPI_CYCLE_DAYS`（KPI）で調整可能。

## コンプライアンス（重要）
- ナビ/ワンキャリア/Indeed等の**スクレイピングは各社ToS/robotsに抵触し得る**ため、本ツールは**取得済み/許諾済みCSVの取り込み・統合・採点**に徹します（系統A/Cは半手動・許諾データ前提）。自動化は公開APIの系統B（gBiz/法人番号）に寄せます。
- 名簿のオプトアウト・特定電子メール法（メールはオプトイン）・個人情報の適正取得を遵守して運用してください。

# 採用担当者「個人名」取得層（Wantedly / ハローワーク）

中堅〜大手中心の母集団（このgBiz由来1000件）は、自社採用ページ・メールに採用担当の**個人名を出さない**（窓口は「採用担当」「recruit@」）。個人名が公開で出るのは**採用広報が活発な媒体**＝Wantedly等。そこを社名で引いて拾う層です。実装は [`src/scrape-names.js`](src/scrape-names.js)（アダプタ）＋ [`src/build-names.js`](src/build-names.js)（駆動）。

```powershell
npm run names                                            # leads-daihyou-1000.csv → sources/A-names.csv
node src/build-names.js --in leads-daihyou-1000.csv --out sources/A-names.csv --limit 200 --concurrency 2
node src/build-names.js --include-experimental           # ハローワーク枠も試す（要セッション較正）
```

- **取得は全て `polite.js` 経由**（robots遵守・ホスト別レート制限・7日キャッシュ）。robots禁止なら `blocked` でスキップし、ゴミを後段へ流さない。
- **氏名の確定は既存資産を再利用**：`extract.js`（採用文脈の正規表現・conf0.7）＋ `jp-names.js`（姓ガゼッティアで`isFullName`検証）＋ 媒体投稿者セレクタ（conf0.5）。**姓辞書で検証できないトークンは人名にしない**（精度優先）。
- 出力 `sources/A-names.csv`（`企業名/法人番号/採用担当者名/役職/部署/取得元媒体/根拠URL/担当者確度/探索結果/取得日`）は **manifest に系統B（肉付け）で登録済み**。`build-list.js` のマージで `採用担当者名` を**確度優先**で採用（系統A扱いにすると全社へ新卒フラグが誤って立つため、あえてB）。

| アダプタ | 状態 | 備考 |
|---|---|---|
| **Wantedly** | 較正済み（静的取得で動作） | エンドポイントは `wantedly.com/projects?q=<社名>`（SSR・robots許可）。投稿者名は `[class*="MemberName"]` から抽出。**会社一致を必須化**し無関係企業の名を出さない。`NAMES_RENDER=auto` でPlaywright描画も可（chromium同時起動はメモリを食うため並列1推奨） |
| **ハローワーク** | experimental | 公的データで低リスクだが現行ネットサービスは**JSF（ViewState付きセッション）**でGET検索が成立しにくい。担当者欄は係名止まりも多い。`--include-experimental` で枠を起動 |

> 純ロジック（`firstFullName`/`extractPersonName`/`pickDetailUrls`/`namesMatch`/`companyOnPage`）は `npm test` で回帰検証済み（ネットワーク不要）。抽出経路は実証済み（例: 「株式会社FLINTERS」→ 投稿者「早瀬峻介」を根拠URL付きで取得）。

### 実測と限界（このgBiz中堅〜大手1000件での歩留まり）
実母集団10社（JFEスチール/住友商事/日産化学/五洋建設等）で試走した結果、**個人名HIT 0%**。`探索結果`は全件 `Wantedly:company-mismatch`＝**検索は募集を返すが、対象の中堅大手はWantedlyの結果に現れない**（Wantedlyの全文検索は社名スコープでなく人気のスタートアップ募集を返す）。これは [採用担当者名が取れない真因＝母集団] の決定的な裏付けで、抽出技術ではなく**母集団の問題**であることを再確認した（会社一致ガードが誤名の混入を完全に防いでいる点も確認）。

> **使いどころ**：本層は**母集団を採用広報の活発なSME/ベンチャーに寄せたとき**に効く（個人名が募集投稿に出るため）。中堅大手中心の現リストでは収量は出ない。媒体セレクタは将来のDOM変更時に `scrape-names.js` の `NAME_ADAPTERS[].authorSel` / `detailSel` を再較正する。

## MOCHICA向けリスト（アポが取れる基準で選定）

「完成リストが今のリストよりアポを取りやすい」と**自信を持って言い切る**ために、汎用ICP採点（`quality.js`）と決別して MOCHICA 専用の **アポ取得期待値モデル** を新設した。

### 確定ターゲット
- **企業像**：新卒を増やしたい中小（従業員 **50〜150名** がスイート）
- **タイミング**：**28卒の媒体選定・採用設計期**を最優先（＝いま採用管理ツールを比較検討するテーブルに乗る瞬間）
- **到達性**：テレアポで“担当者名指し”で繋ぐため、**電話＋採用担当者名**を重視

### 採点ロジック（`src/mochica-fit.js`）
アポ取得期待値 = Σ(次元 × 重み) − ペナルティ。各次元に**確信度**を併走させ、スコアが実データ由来か代理推定由来かを可視化する。

| 次元 | 重み | 中身 |
|---|---|---|
| ① 新卒インテント | 0.34 | 実在性で階級化：**マイナビ実取得 > 新卒フラグ確定 > 求人検索で採用中 > 採用ページのみ(代理)**。複数媒体出稿・募集職種数で痛みの濃さを加点 |
| ② 規模フィット | 0.26 | 50-150名=100。**大企業(自前/競合ATS濃厚)・零細(新卒母数薄い)を強めに減点** |
| ③ 到達性 | 0.20 | 電話妥当＋採用担当者名（名指し架電可）＋部署/役職 |
| ④ タイミング | 0.14 | 28卒設計期係数（6-8月＝サマーインターン準備＋媒体選定の山）＋辞退/更新トリガーでレコード単位に引き上げ |
| ⑤ 継続・信用 | 0.06 | 設立年（毎年新卒を採れる体力）＋補助金 |

ペナルティ：競合ATS導入・大企業・既存顧客・DNC を減点。

### 実行（`npm run mochica`）
```
npm run mochica            # 採点＋確信度レポート＋なぜ今なぜこの企業 付きCSVを出力
npm run mochica:enrich     # 上位社をマイナビ実取得して採用担当者名を補完→再採点
```
- **架電ベースは `leads-telapo-1000.csv`**（電話番号を全件保持）。`leads.master.csv` は電話3/1008・担当者0/1008で**架電不能**だったため、ベースを telapo に切替。`--merge` で daihyou / master を法人番号→社名キーで合流し、連絡先＋新卒インテントを補完する。
- 出力 `leads-mochica.scored.csv` は先頭に **アポ期待度／優先度／確信度／なぜ今なぜこの企業／各次元スコア／採点根拠** を付与。
- コンソールに **確信度レポート**を出す：上位N社のうち「新卒採用を実データで確認」「規模スイート」「電話で架電可能」「採用担当者を名指し可能」の割合。これが“言い切る”ための裏取り。

### マイナビ Playwright 実取得（`npm run mynavi` / `src/scrape-mynavi.js`）
採用担当者名の収集が主軸。実ブラウザ＋人手と同じ**カーソル操作**（mouse.move→click→scroll）で動く。実DOM較正済み（2026-06）：

1. フリーワード検索(GET) `…/searchCorpListByGenCond/index?actionMode=searchFw&srchWord=社名` で結果を取得
2. `corp{ID}/outline.html` を社名一致で特定 → 該当リンクをカーソルでクリック
3. `is.html`（インターンシップ＆キャリア）/ `employment.html`（前年の採用データ）の**『問合せ先』ブロック**から `部署＋採用担当者名＋電話＋メール` を構造抽出
   - 例：`問合せ先 人事総務部 野崎瑠美 03-… saiyo@…` → 全項目取得を実証
4. 氏名は**問合せ先/採用担当の文脈がある面に限定**して抽出。**地名（関東支店・宮城県等）を弾く GEO_SUFFIX ガード**で誤抽出を防ぎ、辞書外の姓は「部署直後」という構造位置で許容する（精度優先：誤った担当者名はゼロ件より悪い）。

```
node src/scrape-mynavi.js "会社名1" "会社名2"      # 単体動作確認
MYNAVI_HEADFUL=1 node src/scrape-mynavi.js "…"     # ブラウザ画面を表示
MYNAVI_DEBUG=1   node src/scrape-mynavi.js "…"     # data/mynavi-cache に html/png を保存（較正用）
```
環境変数：`MYNAVI_GRAD_YEAR=28`（既定・28卒）/ `MYNAVI_POLITE_MS`（社間ディレイ）/ `MYNAVI_NAV_TIMEOUT_MS`。

> **設計上の含意**：マイナビの「問合せ先」は中堅大手でも担当者個人名が載ることがあり、これは「中堅大手は個人名非公開」（自社サイト経路）を**媒体側の別経路で突破**する手になる。
